Base de Données ONU

Conception d'une base de données relationnelle à partir de données démographiques mondiales

SAE 1.04 — 2025 SQL • Base de données relationnelle • CSV

Contexte du projet

Source des données

Conception d'une base de données relationnelle à partir de données démographiques mondiales fournies par l'Organisation des Nations Unies.

Données brutes complexes

Les données, initialement stockées dans des fichiers CSV et XLSX, nécessitaient un important travail de compréhension, de nettoyage et de structuration.

Conception relationnelle

Transformation de données tabulaires en un modèle relationnel normalisé, garantissant l'intégrité et la cohérence pour une exploitation efficace.

Défis du projet

  • Compréhension des données : Analyse approfondie de la structure et du sens des données ONU
  • Normalisation : Identification des entités, relations et dépendances fonctionnelles
  • Qualité des données : Garantie de la cohérence des types et de l'intégrité référentielle
  • Exploitation future : Base solide pour les applications web et analyses statistiques

Ce que j'ai réalisé

01

Conception du schéma relationnel

Participation à la conception complète de la base de données selon un schéma relationnel rigoureux. Identification des entités principales (pays, régions, indicateurs démographiques), définition des relations entre elles, et établissement des clés primaires et étrangères.

02

Traitement et nettoyage des données

Réalisation d'un important travail de traitement des données pour garantir leur cohérence. Conversion des valeurs numériques stockées en chaînes de caractères vers des types appropriés (INTEGER, REAL), gestion des valeurs manquantes et normalisation des formats.

03

Validation par requêtes SQL

Rédaction de requêtes SQL complexes pour vérifier l'intégrité et la cohérence des données. Contrôles de contraintes, validation des relations entre tables, et détection d'anomalies potentielles dans les données importées.

Explications techniques

La conception du schéma relationnel a nécessité une analyse approfondie de la structure des données ONU pour identifier les entités principales et leurs relations.

Entités principales identifiées

Pays (Country)
  • code_pays (PK)
  • nom_pays

Sous Region (subregion)

SQL — Création des tables 
-- Table des pays
CREATE TABLE Country (
    code_pays VARCHAR(3) PRIMARY KEY,
    nom_pays VARCHAR(100) NOT NULL,
    code_subregion VARCHAR(3),
    FOREIGN KEY (code_subregion) REFERENCES subregion(code_subregion)
);

-- Table des sous-régions
CREATE TABLE subregion (
    code_subregion VARCHAR(3) PRIMARY KEY,
    nom_subregion VARCHAR(100) NOT NULL,
    code_region VARCHAR(3),
    FOREIGN KEY (code_region) REFERENCES Region(code_region)
);

-- Table des régions
CREATE TABLE Region (
    code_region VARCHAR(3) PRIMARY KEY,
    nom_region VARCHAR(100) NOT NULL,
    code_continent VARCHAR(2),
    FOREIGN KEY (code_continent) REFERENCES Continent(code_continent)
);

-- Table des données démographiques
CREATE TABLE fact_pop (
    id_donnee SERIAL PRIMARY KEY,
    code_pays VARCHAR(3) NOT NULL,
    year INTEGER NOT NULL,
    value REAL,
    FOREIGN KEY (code_pays) REFERENCES Country(code_pays),
    UNIQUE (code_pays, year)
);

Problématiques rencontrées

Types de données incohérents

Les fichiers CSV contenaient des valeurs numériques stockées comme du texte, empêchant les calculs et comparaisons.

  • Population : "1234567" (texte)
  • Taux de natalité : "12.5" (texte)
  • Présence de séparateurs variables
Conversion et validation

Mise en place d'un processus de conversion systématique avec validation des types.

  • Conversion vers INTEGER ou REAL
  • Gestion des valeurs NULL
  • Validation des plages de valeurs

Plusieurs requêtes SQL ont été rédigées pour vérifier l'intégrité et la cohérence des données après importation.

Vérification de l'intégrité référentielle

SQL — Vérification des références 
-- Vérifier qu'il n'y a pas de pays sans sous-région
SELECT c.code_pays, c.nom_pays
FROM Country c
LEFT JOIN subregion s ON c.code_subregion = s.code_subregion
WHERE s.code_subregion IS NULL;

-- Vérifier qu'il n'y a pas de données orphelines
SELECT d.id_donnee, d.code_pays
FROM fact_pop d
LEFT JOIN Country c ON d.code_pays = c.code_pays
WHERE c.code_pays IS NULL;

Validation des plages de valeurs

SQL — Validation des valeurs 
-- Détecter les valeurs aberrantes de population
SELECT c.nom_pays, d.year, d.value
FROM fact_pop d
JOIN Country c ON d.code_pays = c.code_pays
WHERE d.value < 0 OR d.value > 10000000000;

-- Vérifier la cohérence temporelle
SELECT code_pays, year, COUNT(*) as nb_doublons
FROM fact_pop
GROUP BY code_pays, year
HAVING COUNT(*) > 1;

Requêtes d'analyse statistique

SQL — Analyses 
-- Top 10 des pays les plus peuplés (dernière année)
SELECT c.nom_pays, d.value as population, d.year
FROM fact_pop d
JOIN Country c ON d.code_pays = c.code_pays
WHERE d.year = (SELECT MAX(year) FROM fact_pop)
ORDER BY d.value DESC
LIMIT 10;

-- Évolution de la population mondiale par décennie
SELECT 
    FLOOR(year / 10) * 10 as decennie,
    SUM(value) as population_mondiale
FROM fact_pop
GROUP BY FLOOR(year / 10)
ORDER BY decennie;

Contraintes d'intégrité

Plusieurs mécanismes ont été mis en place pour garantir la qualité des données :

Contraintes NOT NULL

Les champs essentiels (codes pays, années, noms) ne peuvent pas être vides.

Clés étrangères

Empêchent l'insertion de données référençant des entités inexistantes.

Contraintes UNIQUE

Évitent les doublons (ex: un seul indicateur par pays et par année).

Contraintes CHECK

Validation des plages de valeurs acceptables (années, populations positives).

Enseignements clés

Importance de la modélisation

Un schéma relationnel bien conçu est essentiel pour garantir la cohérence et l'exploitabilité des données sur le long terme. La normalisation prévient les redondances et les anomalies de mise à jour.

Qualité des données source

Les données brutes nécessitent presque toujours un nettoyage approfondi. La validation systématique par requêtes SQL permet de détecter et corriger les incohérences avant qu'elles ne posent problème.

Rigueur méthodologique

Le succès d'un projet de base de données repose sur une méthodologie rigoureuse : analyse, conception, implémentation, validation. Chaque étape conditionne la qualité du résultat final.
